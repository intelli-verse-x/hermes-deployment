AI Agency — how it's built

The AI Agency is our productized agent: a self-improving AI worker (built on the open-source Hermes Agent) that does real agency-style work — research, content production, ads, design, game ops — and lives wherever the team does: a desktop app, Discord, Telegram, CLI. This site explains the architecture of each codebase that makes it up, in plain language first and engineering detail second.

The one-paragraph version A pinned fork of the Hermes runtime (hermes-agent) does the thinking; a small ops repo (hermes-deployment) tells it about our world and runs it on the cluster and on laptops; two desktop shells put a friendly window on top; and a shared skills registry (agent-skills) teaches every AI client in the org the same playbooks. Everything talks to the rest of the platform (models, secrets, work tracking, content pipelines) through MCP — the same protocol the admin portal copilot uses.

The four codebases

🤖 hermes-agent
Our fork of the Nous Research agent runtime — the brain. Gateway to Discord/Telegram/CLI, learning loop, MCP tools, and the Hermes Desktop client. Pinned to upstream tags.
🚀 hermes-deployment
The operational glue: pre-wired config, bundled skills, Kubernetes manifests, laptop installer, and runbooks. Day-to-day work happens here, not in the fork.
🖥️ Desktop apps
Two shells: the Tauri "Intelliverse X" app that wraps the hosted admin portal, and Hermes Desktop (Electron) with VPN, MCP and Skills panels.
🧩 agent-skills
The org's skill registry on the agentskills.io open standard — write a playbook once, use it from Hermes, Cursor, Claude Code, Codex and the admin copilot.

Design principles

  • Fork the surface, not the runtime. hermes-agent is pinned to upstream tagged releases; our own code lives in the deployment repo, plugins, and UI shells. Patches worth keeping get PR'd back upstream. This keeps us on the release train of a 200k-star project instead of maintaining a divergent fork.
  • Config over code. Everything org-specific — which model gateway, which MCP servers, which skills — is YAML and markdown in hermes-deployment. Redeploying the agent with new behavior is a config change, not a rebuild.
  • One skill format everywhere. Skills follow the agentskills.io standard, so the same SKILL.md powers Hermes, Cursor, Claude Code and the admin portal copilot.
  • Everything through MCP. The agent has no bespoke integrations. Content Factory, Gas Town, Leantime, n8n, Firecrawl, Google Stitch — all reached through the same Model Context Protocol servers the admin copilot uses. Add an MCP server and every agent in the org gets the capability.
  • All model traffic through LiteLLM. One gateway, one budget, automatic Langfuse tracing. The agent never holds a raw provider key.
