hermes-deployment — the ops glue

DAY-TO-DAY REPO   intelli-verse-x/hermes-deployment. The runtime fork is the binary substrate — this repo is where the AI Agency actually gets configured, deployed and operated. If you change how the agent behaves, which tools it has, or where it runs, you change this repo.

What lives here

PathWhat it is
config/config.yamlThe pre-wired Hermes config: LiteLLM as the (only) model provider, MCP servers for Gas Town/Beads, Leantime, Content Factory, n8n and Firecrawl, memory settings, and 7 external skill-repo mounts so we never duplicate a skill.
config/AGENTS.mdThe context file Hermes auto-loads at the top of every session — a map of the whole intelli-verse-x stack, so the agent knows what every service is and which tool reaches it.
config/.env.exampleSecrets template (minimum: LITELLM_API_KEY).
skills/Deployment-bundled skills: ivx-stack-tour, ivx-products-tour, ivx-content-factory, ivx-gastown-bridge.
k8s/Cluster manifests for the always-on gateway pod: deployment + service + ingress + configmap.
scripts/install-local.shOne-command laptop install — every engineer gets the identical agent locally.
scripts/entrypoint.shK8s entrypoint that seeds the persistent volume from the bundled config on first boot.
scripts/seed-cluster-secrets.shPopulates the pod's secrets from existing cluster secrets where possible.
DockerfileProduction image: official Hermes (from our pinned fork) + our binaries + the config bundle.
docs/Operating runbook, cited best-practices — and this architecture site (docs/architecture-site/).

Deploying

# build + push the image
docker buildx build --platform linux/amd64 \
  -t 970547373533.dkr.ecr.us-east-1.amazonaws.com/hermes-agent:latest --push .

# seed the config bundle + secrets
kubectl create configmap hermes-config-bundle -n aicart \
  --from-file=config.yaml=config/config.yaml \
  --from-file=AGENTS.md=config/AGENTS.md \
  --dry-run=client -o yaml | kubectl apply -f -
./scripts/seed-cluster-secrets.sh

# roll out
kubectl apply -f k8s/
kubectl rollout status -n aicart deploy/hermes

Local install (laptop)

git clone git@github.com:intelli-verse-x/hermes-deployment.git
cd hermes-deployment && ./scripts/install-local.sh
# edit ~/.hermes/.env  (at minimum: LITELLM_API_KEY)
hermes doctor && hermes

Skill mounts — why 7 repos

The config mounts external skill directories read-only; Hermes sees every skill in those trees as if it were local, and its own agent-curated skills (in ~/.hermes/skills/) shadow them on name collisions. This is how one skill, written once, serves every agent:

  1. intelli-verse-x-agent-skills/skills — the org registry (canonical)
  2. Intelli-verse-X-SDK/skills — game-SDK, AI-integration, cross-platform skills
  3. agent-skills/.agents/skills — the Anideebee mega-collection (opt-in)
  4. Agentic-SEO-Skill/resources/skills — SEO playbooks
  5. geo-optimizer-skill/…/skills — GEO (AI-search optimization) playbooks
  6. hyperframes/skills — 3D/AR/VR skills
  7. hermes-deployment/skills — the 4 bundled deployment skills above

How Hermes plugs into the 5-layer stack

LayerServiceHow Hermes uses it
1 · StrategyLeantimemcp_leantime_* — list/create/update tickets; strategy lives here.
1 · StrategyGitHubgh CLI in shell tools (no MCP needed).
2 · CoordinationBeads (bd)mcp_gt_bd_* — every nontrivial request becomes a bead; auditable work graph.
2 · CoordinationBernsteinHermes dispatches via gt sling; Bernstein owns each project's coding crew.
3 · Multi-projectGas Townmcp_gt_* — rigs, mail, nudges, convoy status across all repos.
4 · PlatformLiteLLMAll LLM calls (provider: custom).
4 · PlatformInfisicalOptional secret backend via machine identity token.
4 · PlatformLangfuseAutomatic — LiteLLM forwards traces; zero Hermes config.
5 · Gluen8nmcp_n8n_* — workflow triggers (gt-escalate → Discord, etc.).
5 · GlueGitHub ActionsVia gh CLI; cross-provider review workflow.
The mental model: beads are the unit of work Per config/AGENTS.md, every nontrivial request the agent takes becomes a bead (bd create → work → bd close), even when the agent does the work itself — so everything the AI Agency does is auditable. Work that belongs to a specialist coding crew is slung to a Gas Town rig instead of done inline.
