hermes-agent — the runtime fork

FORK · PINNED TO UPSTREAM   intelli-verse-x/hermes-agent, forked from NousResearch/hermes-agent (MIT, ~200k stars). This is the brain of the AI Agency: the agent loop, the multi-platform gateway, the learning system, and the Hermes Desktop client all live in this one monorepo.

What it is, in plain language

Hermes is "the agent that grows with you". Unlike a chatbot, it has a closed learning loop: after finishing a complex task it writes itself a skill, improves that skill the next time it's used, keeps searchable memory of every past conversation, and builds a model of each user across sessions. It runs 24/7 on a server — you talk to it from Discord or Telegram while it works on a cloud machine, and pick the same conversation up from your laptop later.

Why we forked (and what we don't do)

  • Reproducible builds — our Docker image pins a specific upstream tag, so cluster rollouts can't drift when upstream moves (they release fast).
  • In-house plugins — org-specific extensions (e.g. a memory provider reading our Beads work graph) live in the fork; generic fixes get PR'd back upstream.
  • Patch independence — we can ship a security fix without waiting on upstream's merge cycle.
What we deliberately don't do We do not rewrite the runtime. The fork tracks upstream closely; anything org-specific that can live in hermes-deployment (config, skills, manifests) lives there instead. "Fork the surface, not the runtime."

The capabilities that matter for the Agency

Learning loop

Creates skills from experience, self-improves them during use, FTS5 full-text search over its own past sessions, and Honcho dialectic user modeling. Compatible with the agentskills.io standard — which is why our shared skill registry plugs straight in.

MCP-native tools

Every integration is an MCP server — the same ones the admin portal copilot uses (Content Factory, Gas Town, Leantime, n8n, Firecrawl…). No bespoke connectors to maintain.

Lives where you do

One gateway process serves Discord, Telegram, Slack, WhatsApp, Signal, Email and CLI at once. Start a conversation on Discord, continue it in the terminal.

Runs anywhere

Six terminal backends — local, Docker, SSH, Singularity, Modal, Daytona. Serverless backends hibernate when idle, so a 24/7 agent costs nearly nothing between tasks.

Delegation

Spawns isolated subagents for parallel workstreams, and can write Python scripts that call its own tools via RPC — collapsing multi-step pipelines into single turns.

Scheduled automations

Cron in natural language: "every weekday at 9am, summarize new Leantime tickets to Discord" — no external scheduler needed.

Model access

The runtime supports any provider, but in our deployment every LLM call goes through LiteLLM (configured as provider: custom in hermes-deployment's config.yaml). That gives us one budget, one key to rotate, model switching without code changes, and automatic Langfuse tracing. The agent never sees a raw OpenAI/Anthropic key.
